# TON Wallet — Self-Custodial Testnet Wallet

Web-приложение криптокошелька для TON testnet без собственного backend.

## Запуск

```bash
npm install
npm run dev
```

Приложение будет доступно на `http://localhost:5173`.

Для production-сборки:

```bash
npm run build
npm run preview
```

## Тесты

```bash
npm run test        # watch-режим
npm run test:run    # один прогон (CI)
npm run test:coverage
```

Покрытие (см. `npm run test:coverage`): **`wallet.ts`**, **`tonApi.ts`**, **`formatAddress.ts`** — 100% строк; **`useWallet`** ~96%; **`spoofing`** ~98%. **`TonClient`** подменяется в тестах через `injectTonClientForTests` / `resetTonClientToDefault`. В jsdom-тестах хука мокаются `createWallet` / `loadWalletFromMnemonic` / `importWallet`, чтобы не гонять `@ton/crypto` в браузерном окружении. Smoke/UX: **Welcome**, **Dashboard**, **Send**, **Receive** (в т.ч. копирование адреса).

- Фикстура валидной мнемоники: `src/test/fixtures/valid-mnemonic.json` (для сценариев с `saveMnemonic` в тестах хука).
- UI-тесты: `// @vitest-environment jsdom`; модульные тесты крипто/TON API — **Node**, алиас `buffer` → `node:buffer` при `mode === 'test'` в `vite.config.ts`.

## Функционал

- **Создание кошелька** — генерация 24-словной мнемонической фразы, отображение с требованием подтверждения сохранения
- **Импорт кошелька** — восстановление по мнемонической фразе с валидацией
- **Dashboard** — адрес, баланс (авто-обновление каждые 15 сек), история транзакций с поиском
- **Receive** — полный адрес, QR-код, копирование в буфер обмена
- **Send** — форма отправки с валидацией, подтверждение, обратная связь о статусе
- **Защита от подмены адреса** — система предупреждений при отправке

### Ключевые решения

**Без backend.** Все операции выполняются на клиенте. Взаимодействие с блокчейном — через публичный TON Center HTTP API (`testnet.toncenter.com`).

**WalletContractV4.** Выбран как стандартный контракт кошелька TON, поддерживаемый всеми основными клиентами.

**Polling.** Баланс и транзакции обновляются каждые 15 секунд по таймеру. В production-решении стоит использовать WebSocket/SSE подписку через TON API v3 или tonapi.io.

**Хранение мнемоники.** Мнемоника хранится в `localStorage` в открытом виде. Это осознанный компромисс для тестового задания. В production необходимо:
- Шифрование мнемоники паролем пользователя через Web Crypto API
- Хранение зашифрованных данных в IndexedDB
- Автоматическая блокировка после таймаута

Текст условия — в файле [`testovoe.md`](./testovoe.md). Ниже — то, что в ТЗ просили описать в README: **архитектура**, **компромиссы**, **логика проверок** и **что бы доработал**.

### Компромиссы (зачем так сделано)

В ТЗ сказано: **без своего backend**, **только testnet**, данные для UX **можно хранить локально в браузере**, и **не нужна production-безопасность**. От этого и отталкивался.

- **Нет сервера** — по условию. Всё на клиенте, к сети ходим через публичный HTTP API testnet.
- **Мнемоника в `localStorage` как есть** — это слабо для «настоящего» кошелька, но для тестового задания и явного пункта про «не production» — нормальный компромисс: меньше кода, проще проверить сценарии. В бою нужны пароль, шифрование, таймаут блокировки.
- **Обновление баланса и истории раз в 15 секунд** — простой способ не усложнять проект WebSocket’ами и ключами к сторонним API. Для демо testnet хватает.
- **Книга «уже отправлял сюда» в `localStorage`** — локально, без backend, как разрешено в ТЗ. Помогает и UX (повторные переводы без лишнего шума), и проверкам при подмене адреса.
- **QR на Receive** — в ТЗ как бонус; добавил для удобства, на обязательный минимум не влияет.
- **Тесты** — в тестах React-хука крипту мокаю, чтобы в jsdom не тащить тяжёлые вызовы; сами `wallet` / `tonApi` тестируются отдельно в Node.

### Архитектура и стек (почему такой выбор)

- **Папки `services` / `screens` / `utils`** — логика кошелька и сеть отдельно от экранов: проще читать код и писать тесты без UI.
- **React + TypeScript + Vite** — привычный стек для небольшого SPA, быстрый запуск и сборка.
- **`@ton/ton`, `@ton/core`, `@ton/crypto`** — официальные библиотеки под TON, меньше ручной возни с форматами.
- **Контекст + `useWallet`** — одно место, где лежит состояние кошелька; экраны не передают друг другу десяток пропсов.
- **react-router-dom** — отдельные страницы (главная, отправка, получение) и редиректы «есть кошелёк / нет».
- **Tailwind** — быстро собрать **минималистичный** интерфейс; предупреждения при отправке сделаны **заметными** (модалка, цвета, для «опасных» сценариев — чекбокс), как в ТЗ.
- **Vitest + Testing Library** — те же настройки, что у Vite; проверяются и сервисы, и основные экраны.

### Логика проверок перед отправкой

Кратко: сравниваем введённый адрес с **своим** и с **адресами, на которые уже отправляли** (локальная история). Новый адрес — предупреждение «проверьте». Адрес **похож в начале/конце** на уже известный, но **не совпадает целиком** — сильное предупреждение (типичная подмена «похожего» адреса). Отправка **самому себе** — отдельное предупреждение. Подробные уровни и оформление — в разделе **«Защита от подмены адреса»** ниже по файлу.

## Защита от подмены адреса (Address Spoofing)

Реализована многоуровневая защита:

1. **Книга известных адресов** — адреса, на которые уже отправлялись средства, сохраняются в localStorage. При повторной отправке на тот же адрес предупреждения не показываются.

2. **Предупреждение «Новый адрес»** (severity: medium) — если адрес не найден в истории отправок, показывается предупреждение о необходимости проверки.

3. **Детекция визуально похожих адресов** (severity: high) — если введённый адрес имеет одинаковые первые или последние 6 символов с известным адресом, но отличается в середине — показывается блокирующее предупреждение. Это защищает от классической атаки, при которой злоумышленник генерирует адрес, визуально похожий на целевой.

4. **Отправка на свой адрес** (severity: high) — предупреждение при попытке отправить средства самому себе.

Все предупреждения с высоким уровнем риска требуют **явного подтверждения** через чекбокс перед отправкой. Предупреждения отображаются в модальном виде с яркой цветовой кодировкой (красный для high, оранжевый для medium).
